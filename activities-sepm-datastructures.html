UNIT 3: DATA STRUCTURES REFLECTION

Read Dicheva & Hodge (2018). Think about an online system which you use on a daily basis. Consider how it might operate at the back-end using data structures.

RESPONSE

My answer is based on leveraging my understanding of data structures as outlined in “Active Learning through Game Play in a Data Structures Course”, Dicheva & Hodge (2018) to demonstrate my awareness of how data structures are foundational to the operation of everyday systems.

Before approaching this answer, I had to recap my understanding and knowledge of Data Structures. I completed two courses on LinkedIn Learning Platform on Programming Foundations: Data Structures and Learning Java. I then approached the answer and broke it down into the following steps:

Step 1: Considering online systems, I use daily

  • Email Service - I use different types of email services. At work, I use Outlook and Yahoo for personal use.
  • Social Media - throughout the day catching up with news, friends and family using Facebook, Instagram and Twitter
  • Streaming platforms - after a long day of work and doing assignments, I stream before going to bed on Netflix of Sky
  • E-commerce site - to order for my family on platforms like Marks & Spencer, Currys or B&Q

    • Step 2: Identify Core Operations of the System in relation to Data Structures

    When I work on my emails, the core operations that I use of that email service are:

  • Managing my inbox (viewing, sorting, deleting or removing)
  • Sending and receiving emails
  • Searching through old emails
  • Labelling and categorising my emails

    • Step 3: Mapping Back-end Operations Data Structures

    Looking and thinking how data structures are used in the back end to implement the functionality of the email service I thought through the different functionalities

  • Email inbox management:
      • Queue Data Structure - emails are stored in a queue (FIFO structure), ensuring that the newer emails are added at the end and older emails are processed first when viewed.
        List Data Structure – is necessary and used when sorting and searching operations
    • Searching emails:
        • Hash Map Data Structure – maps the keywords or sender names to email IDs to enable quick lookups.
          Binary Search Tree Data Structure – is used to keep these emails sorted by date, sender or subject line for fast retrieval based on the criteria used.
      • Labelling or categorizing emails:
          • Tree Data Structure – used when organizing emails under categories where the search folder is a node; subfolders can be children of that node.
            Graph Data Structure – is used if the emails can belong to multiple categories or labels (with edges representing connections between categories and emails).
        • Sending and receiving emails:
            • Queue Data Structure – sending and receiving emails is handled by queues on the server side, where each email is processed In the order it's received (FIFO)

          Step 4: The Relationship Between Data Structures and Functionality

          This is based on linking each operation to the appropriate data structure to show how that structure supports the operation efficiently e.g.
        • If you think about how the inbox is used – it uses queues to ensure that the emails are displayed in the order they were received. This means we see the newest email first, while order emails are accessible but further down the queue.
        • The search functionality has a hash map and enables fast keyword lookup, allowing users to find emails by sender or subject quickly.

          • Step 5: Data Structure learning

          Finally, my conclusion based on Active Learning through Game Play in a Data Structures Course, Dicheva & Hodge (2018) is to aid problem-solving e.g.

        • Stacks – help us understand how a history of actions (such as undo/redo features) might be implemented in an email client
        • Queues and Trees - aids in visualising how emails are categorised and managed efficiently.
